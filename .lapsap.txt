Capital Bridge Suite — Monorepo summary (for developers)
What it is
capitalbridge-suite is an npm workspaces monorepo (apps/*, packages/*) for the Capital Bridge product: auth + billing on login.*, main advisory hub on platform.*, specialised model apps on their own subdomains, optional API app on api.*, and Supabase as DB + Auth. Deployments are multiple Vercel projects, one per app, same Git repo.

Folder map
Path	Role
apps/login	Next.js: pricing, signup, login, checkout, payment return, forgot/reset password, API routes under /api/* (proxies + webhooks). Domain: login.thecapitalbridge.com.
apps/platform	Next.js: advisory dashboard at / (membership gate), /login redirect to login app. Domain: platform.thecapitalbridge.com.
apps/api	Next.js API routes: billing authority (/billing/*), Billplz webhook, confirm-payment, request-bill. Domain: api.thecapitalbridge.com.
apps/capitalstress, capitalhealth, incomeengineering, forever	Next.js “tool” apps; each has dashboard, uses shared Supabase + login redirect.
packages/db-types	TypeScript Database types for Supabase public tables used in code (not a full introspected schema).
packages/supabase	Browser / server / service_role Supabase clients.
packages/advisory-graph	Persona, entitlements, plan helpers, shared graph utilities.
packages/shared	Shared constants (e.g. LOGIN_APP_URL).
packages/auth, packages/membership, packages/ui	Auth/session/membership helpers and UI stub.
supabase/migrations	SQL migrations (plans, billing_sessions, payments, RPCs, RLS, etc.).
docs/	Extra docs (e.g. live test mode).
Root README.md and ARCHITECTURE.md describe flows; VERCEL_MONOREPO.md + VERCEL-SETUP.md cover Vercel + env.

Supabase schema (conceptual)
Source of truth for money + access is Postgres public (plus auth.users).

Object	Purpose
plans	Catalog: slug (trial, monthly, quarterly/sku names), price_cents, duration_days, is_trial.
profiles	id = auth user; email, trial_use_count, payment_status, pending_plan.
memberships	user_id, plan_id, status (pending → active/expired/…), dates, billing_session_id, expiry columns.
payments	billplz_bill_id (idempotency), membership_id, optional user_id/plan_id, amounts, provider, raw_webhook.
billing_sessions	Payment-first checkout: email, plan_id, user_id (pre-created user), bill_id, status (pending / bill_created / paid), membership_id.
pending_bills	Alternate payment-first path before session model.
active_memberships	View (read-only): one logical row per user with plan (slug), start_date, end_date — used by platform (and similar) to decide access.
billing_events	Audit / debug events (webhook, payment, etc.).
payment_events	Events tied to billing_session_id.
Relationships (high level)

auth.users ↔ profiles.id, memberships.user_id, payments.user_id.
plans.id ↔ memberships.plan_id, billing_sessions.plan_id, payments.plan_id.
billing_sessions ↔ memberships.billing_session_id, payments (via bill id / session, depending on flow).
payments.membership_id ↔ memberships.id after activation.
RPCs (important)

activate_membership — manual/admin style (creates/links payment in some migrations).
activate_membership_for_billplz — Billplz path: assumes payments row exists for billplz_bill_id, creates/updates membership, links payment; idempotent when membership_id already set.
Apply whatever migrations exist in supabase/migrations so prod matches code.
Vercel
One Vercel project per app; each uses Root Directory apps/<name> (see VERCEL_MONOREPO.md).
installCommand often cd ../.. && npm install so workspace packages resolve.
Env: each app needs NEXT_PUBLIC_SUPABASE_*; login/api need Billplz, SUPABASE_SERVICE_ROLE_KEY where server creates users or calls webhooks; API_APP_URL / NEXT_PUBLIC_API_BASE_URL for proxies.
Cookies: configure .thecapitalbridge.com where needed so subdomains share session.
User-visible outputs (login app screens)
Route (login app)	What the user sees
/	Home / entry.
/pricing	Plan cards / marketing pricing.
/checkout?plan=…	Email + name → pre-creates Auth user (pending) → redirect Billplz.
/payment-return	Return from Billplz; may trigger confirm-payment fallback.
/payment-handoff	Handoff UX around payment URL.
/confirm-payment	Confirm step in older flow.
/login	Sign in; redirect to platform after success.
/signup	Sign-up UI.
/forgot-password	Request reset email.
/reset-password	Set password from recovery link (Supabase email); then redirect toward platform.
Platform (apps/platform): / = dashboard (welcome, tiles or placeholder); /login sends users to login app; middleware enforces auth for /.

Model apps: each has /dashboard (and similar); unauthenticated users redirect to login.thecapitalbridge.com with redirectTo.

Request flows (two big patterns)
Logged-in billing (/billing/create on API): pending membership + payment, Billplz, webhook updates payment and activates membership.
Payment-first (/billing/request-bill): Auth user + billing_sessions + Billplz; API webhook (or login forward) runs payment insert + activate_membership_for_billplz, profile active, recover email for set-password.
